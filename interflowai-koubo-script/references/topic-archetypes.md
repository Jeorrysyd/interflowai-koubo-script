# 口播类型模型

先选一个主模型，再决定是否需要一个辅助模型。不要为了显得完整，把四种模型都塞进一条口播。

## 1. 观点判断型

适合：

- 行业趋势、职业变化、商业判断、创始人观察。
- 用户还没形成判断，需要你帮他看清变化。

核心结构：

```text
强判断 -> 真实证据 -> 拆成 2-3 个原因 -> 对目标用户意味着什么 -> 行动建议或商业桥
```

开头公式：

```text
我最近越来越确定一件事：___。真正拉开差距的不是___，而是___。
```

证据要求：

- 至少一个外部变化、真实现场、客户对话或业务案例。
- 不能只讲宏观趋势。

常见错误：

- 先解释概念。
- 堆自我定位。
- 观点很大，但没有一个真实例子。

## 2. 流程干货型

适合：

- 工具、方法、模板、SOP、工作流、效率提升。
- 用户有明确任务，想知道怎么做。

核心结构：

```text
结果前置 -> 旧流程痛点 -> 新流程三步 -> 每步的输入/AI/人/产物 -> 可复用方式
```

开头公式：

```text
我用 AI 把___这件事，从___变成了___。
```

证据要求：

- 旧流程的痛点或耗时。
- 新流程的真实输出、界面、表格、清单或前后对比。

常见错误：

- 先介绍工具背景。
- 把教程写成说明书。
- 只讲 AI 做什么，不讲人怎么判断。

## 3. 活动复盘型

适合：

- 会议、展会、工作坊、路演、客户拜访、行业现场。
- 用户需要从现场热闹里看出真正变化。

核心结构：

```text
洞察先行 -> 一个现场细节 -> 它说明了什么变化 -> 对目标用户的影响 -> 下一步承接
```

开头公式：

```text
我这次去___，真正让我意外的不是___，而是___。
```

证据要求：

- 现场照片、议程、对话、问题、反馈或会后合作线索。
- 必须说清楚这个现场留下了什么资产。

常见错误：

- 按时间线流水账。
- 从“我今天去了哪里”开始。
- 把活动热闹当成结论。

## 4. 产品方法论型

适合：

- 发布一个系统、工作台、方法论、内容生产线或产品化能力。
- 用户需要理解“这不是一个工具，而是一套方法”。

### 4.1 发布说明型

适合正式介绍一个系统、方法论或工作台。

核心结构：

```text
结果/反差前置 -> 过去问题 -> 新做法 -> 方法拆解 -> 下一步如何复用
```

开头公式：

```text
___这件事，其实不是靠灵感，而是可以被拆成一套工作流。
```

### 4.2 示范工作流型

适合宣传一个 Skill、Agent、浏览器插件、工作台或工具。默认优先使用这个结构，尤其当用户想写“我做了一个 Skill / 我怎么用这个工具”时。

核心结构：

```text
结果或痛点 Hook -> 一句身份介绍 -> 3-4 个动作步骤 -> 每步先讲用户任务再讲工具/Skill 怎么帮忙 -> 价值观收束 -> 下一条预告或评论问题
```

开头公式：

```text
现在很多人想___，但真正卡住的不是___，而是___。
```

步骤写法：

```text
第一步：先解决用户眼前的任务。
不要先介绍功能，先说为什么这一步会卡住。
然后再说 Skill / 工具在这一步帮你做什么。
```

证据要求：

- 系统界面、样本表、输出报告、流程截图、真实结果。
- 至少一个前后对比、真实场景、评论需求或失败样本。
- 解释这个方法如何迁移到别人的场景。

常见错误：

- 炫技。
- 一上来讲技术架构。
- 只讲自己做了什么，不讲用户为什么需要。
- 把产品/Skill 宣传稿写成功能清单，而不是示范一个真实工作流。
- “大家好我是...”放在第一句，导致前 5 秒没有痛点、结果或冲突。

## 选择规则

- 有反常识观点或行业判断：优先观点判断型。
- 有步骤和方法：优先流程干货型。
- 来自真实现场：优先活动复盘型。
- 在讲一个系统或方法论产品：优先产品方法论型。
- 在宣传 Skill、Agent、浏览器、工作台或工具：优先产品方法论型里的示范工作流型，不要写成工具说明书。
- 两类都像时，先问“用户看完最想得到什么”：判断、步骤、现场洞察，还是方法论。
